## Основные принципы WCAG 2.1/2.2

Принципы WCAG 2.1/2.2 основаны на четырех фундаментальных столпах:
Воспринимаемость (Perceivable),
Управляемость (Operable),
Понятность (Understandable) и Надежность (Robust),
или POUR,
обеспечивая доступность контента для людей с разными видами нарушений,
включая когнитивные, двигательные и нарушения зрения. WCAG 2.2 является развитием 2.1, добавляя новые критерии успеха,
особенно для мобильных устройств и пользователей с когнитивными нарушениями, оставаясь при этом обратно совместимым.


Четыре принципа WCAG (POUR):

* Воспринимаемость (Perceivable): Пользователи должны иметь возможность воспринимать информацию и элементы интерфейса, представленные на сайте, вне зависимости от своих сенсорных способностей.
  Пример: Предоставление текстовых альтернатив изображениям для слабовидящих.


* Управляемость (Operable): Интерфейс и навигация должны быть доступны и управляемы с помощью различных методов ввода, не только мыши (например, клавиатуры).
  Пример: Возможность полного управления сайтом с клавиатуры, управление таймерами, предотвращение эпилептических приступов.


* Понятность (Understandable): Информация и управление должны быть понятными, а интерфейс — предсказуемым.
  Пример: Четкий язык, предсказуемая навигация, помощь при заполнении форм, понятные сообщения об ошибках.

* Надежность (Robust): Контент должен быть достаточно надежным, чтобы его мог интерпретировать широкий спектр пользовательских агентов, включая вспомогательные технологии (скринридеры).
  Пример: Правильное использование HTML и семантики, чтобы скринридеры могли корректно «читать» страницу. 

## Практики, которые я применяю при вёрстке/разработке

1. Предоставление текстовых альтернатив изображениям для слабовидящих
2. Возможность полного управления сайтом с клавиатуры
3. Понятные сообщения об ошибках
4. Помощь при заполнении форм
5. Правильное использование HTML и семантики, чтобы скринридеры могли корректно «читать» страницу

## Как ты тестируешь доступность в проекте?

1. Автоматизированное тестирование
   Инструменты:
   axe DevTools (браузерное расширение или интеграция в CI)
   → Проверяет ~50–60% критериев WCAG автоматически.
   Lighthouse (вкладка Accessibility в DevTools)
   → Быстрая оценка + рекомендации.
   eslint-plugin-jsx-a11y (для React)
   → Ловит ошибки на этапе разработки (например, отсутствие alt, неправильные ARIA-роли).
   Pa11y, Accessibility Insights, WAVE
   ⚠️ Автоматизация покрывает не более 60% проблем. Остальное — только ручное тестирование.

2. Ручное тестирование
   a) Навигация с клавиатуры
   Можно ли добраться до всех интерактивных элементов с помощью Tab / Shift+Tab?
   Есть ли видимый фокус (:focus-visible)?
   Работают ли модальные окна, выпадающие меню, табы без мыши?
   b) Скринридеры
   Тестирую с реальными скринридерами:

NVDA + Firefox (Windows, бесплатно)
VoiceOver (macOS / iOS)
JAWS (платный, но часто используется)
Проверяю:

Читается ли содержимое логично?
Объявляются ли изменения (например, после AJAX-запроса)?
Корректно ли работают ARIA-атрибуты (aria-expanded, aria-live, role и т.д.)?
c) Цветовой контраст
Использую инструменты: axe, WebAIM Contrast Checker.
Минимальный контраст:
4.5:1 для обычного текста (WCAG AA)
3:1 для крупного текста (>18pt или >14pt bold)
d) Увеличение масштаба
Проверяю, не ломается ли макет при увеличении до 200–400% (WCAG 1.4.10 Reflow).
